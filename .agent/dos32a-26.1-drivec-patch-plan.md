This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `.agent/PLANS.md` from the repository root. A future contributor should be able to restart from this file alone, with the current working tree and the local DOSBox-X game tree.

## Purpose / Big Picture

The user wants a redistributable zip that updates every non-reserved game under `C:\DOSBox-X\drivec` to the current DOS/32A 26.1 branch output, plus a batch file that can quickly regenerate the same updates later on real hardware. Some games use a separate `DOS4GW.EXE` extender file; others have the extender bound into the game executable and need SB/32A to rebind from an original `.BAK`.

After this work, the user should have a zip file containing only the changed replacement files, an apply batch, and a future update batch that assumes the DOS/32A source binaries are installed at `C:\DOS32A\BINW`.

- Observation: The 26.1 branch still produced binaries that printed `26.0`.
  Evidence: The first SB inventory log began with `SB/32A -- Bind Utility version 26.0`; after updating the version identifiers and rebuilding, the same log reported `SB/32A -- Bind Utility version 26.1`.

  Rationale: It is not a game target and should not be patched or included as a game replacement.
  Date/Author: 2026-05-23 / Codex

- Decision: The future-update batch will assume `C:\DOS32A\BINW\DOS32A.EXE` and `C:\DOS32A\BINW\SB.EXE`.
  Rationale: The user explicitly provided that future source location. External extender slots can be updated by copying `DOS32A.EXE`; bound executables can be regenerated from `.BAK` originals with `SB.EXE /R /O /BN...`.
  Date/Author: 2026-05-23 / Codex

- Decision: Update all active version identifiers to 26.1 before packaging.
  Rationale: The user asked for a 26.1 package, but the rebuilt SB utility and extender still reported 26.0. Packaging those binaries would make the redistributable ambiguous and would not match the branch purpose.
  Date/Author: 2026-05-23 / Codex

- Decision: In folders with standalone `DOS4GW.EXE` (`BIGRED`, `BLOOD`, `REDNECK`, and `RIDES`), patch only `DOS4GW.EXE`.
  Rationale: Those games can use DOS/32A through the external extender slot. Their application executables contain external-loader stubs that SB reports as unknown bound LE files, but rebinding them directly would be a larger behavioral change than needed.
  Date/Author: 2026-05-23 / Codex

- Decision: In eligible folders without standalone `DOS4GW.EXE`, rebind every SB-classified bound executable.
  Rationale: Without an external extender slot, the protected-mode launcher or tool must carry DOS/32A in its own executable. Rebinding all SB-classified bound executables covers games, setup programs, render variants, editors, and expansion launchers while skipping reserved folders.
  Date/Author: 2026-05-23 / Codex

- Decision: Write SB output to `D32TMP.EXE` and copy the temporary file over the target only after SB succeeds.
  Rationale: This avoids corrupting an active executable if a rebind fails or DOSBox-X is interrupted midway. Temporary files are removed after each target.
  Date/Author: 2026-05-23 / Codex

## Context and Orientation

SB/32A is the DOS/32A bind utility. Binding means replacing a DOS extender stub inside a protected-mode executable. Some games keep the extender in a separate file named `DOS4GW.EXE`; for those, replacing `DOS4GW.EXE` with DOS/32A is enough. Other games have the extender bound into the main game `.EXE`; for those, the package must rebuild the game executable from its original `.BAK`.

## Plan of Work

First, inventory all non-reserved directories under `C:\DOSBox-X\drivec`. For each directory, record whether it has a separate `DOS4GW.EXE`, whether a matching `.BAK` exists for an executable, and whether the current executable already appears to be DOS/32A-bound.

Second, patch live game folders. For external `DOS4GW.EXE` slots, copy the current `binw\dos32a.exe` to that path. For bound executables, run SB/32A under DOSBox-X from a temporary staging directory and rebind from the `.BAK` source when a matching `.BAK` exists, otherwise from the active `.EXE`. Do not overwrite `.BAK` originals.

Third, stage the changed files in a package directory. The package must preserve game-relative paths, for example `DOOM2\DOOM2.EXE` or `REDNECK\DOS4GW.EXE`. Add `APPLY-PATCH.BAT` to copy the staged files into place from the zip root. Add `UPDATE-FROM-DOS32A.BAT` to rebuild or copy future updates from `C:\DOS32A\BINW`.

Fourth, compress the package directory into a zip and verify its file list. Keep the package under a local `dist` directory in the repository unless a better path emerges during implementation.

## Idempotence and Recovery

Do not delete `.BAK` originals. External `DOS4GW.EXE` files may be overwritten only when the folder is eligible; their `.bak` files remain the recovery path. Bound executables should be regenerated from `.BAK` originals. Scratch files and batches must start with `D32` and be removed after the run.

If a DOSBox-X SB batch fails midway, stop, restore any active `DOS4GW.EXE` from its `.bak` if it was temporarily changed outside an eligible update, remove scratch files, and record the failure in this plan.

## Purpose / Big Picture

Screamer 2 (`S2`) and Screamer Rally (`SR`) are real DOS/4GW games staged under `C:\DOSBox-X\drivec`. Their setup programs reportedly work with DOS/32A, but the actual games crash on launch. The immediate goal is failure analysis: reproduce the setup-versus-game difference, identify the failing behavior, and decide whether it is a low-risk DOS/32A 26.1 compatibility fix or should be deferred.

If the failure is approachable, this plan will be updated with the exact code change and validation. If it is not approachable, the deliverable is a clear record of what fails, why it appears broad or risky, and the exact cleanup performed.

This plan was reopened on 2026-05-24 after real-hardware evidence showed a DOS/32A kernel IRQ fault in the 65k-color launch path. The earlier DOSBox-X-only conclusion is retained as historical context, but it is no longer the active decision.

## Surprises & Discoveries

- Observation: The active S2/SR `DOS4GW.EXE` files are not currently DOS/32A replacements.
  Evidence: Both are 265396 bytes and contain `DOS/4GW` and `RATIONAL` strings. The current repository `binw\dos32a.exe` is 28368 bytes and contains `DOS/32A` and `SUNSYS` strings.

- Observation: S2 and SR default to the high-resolution executable.
  Evidence: `S2.BAT` and `SR.BAT` each contain only `START65H.EXE`.

- Observation: The default S2 and SR game launch failures are not DOS/32A-specific in the current DOSBox-X configuration.
  Evidence: With original DOS/4GW restored, both `S2.BAT` and `SR.BAT` print `Detection System Failed To Find Hardware` and `Invalid Driver ID`, matching the DOS/32A replacement runs.

- Observation: Mounting the configured `E:` CD/source drive did not change the default launch outcome.
  Evidence: With `mount e C:\DOSBox-X\drivec\S2 -t cdrom` or `mount e C:\DOSBox-X\drivec\SR -t cdrom`, original DOS/4GW and DOS/32A still reached the same hardware-driver failure.

- Observation: SR's setup executable divide-by-zero is not DOS/32A-specific.
  Evidence: `ST.EXE` under DOS/32A reports `DOS/32A run-time (6001): exception 00h` at unrelocated crash address `1:0002D0A3`; under original DOS/4GW it reports `DOS/4GW error (2001): exception 00h (divide by zero)` with the same unrelocated crash address `1:0002D0A3`.

- Observation: SR 3D/3DFX variants are not a clean DOS/32A regression.
  Evidence: Under DOS/32A, `STRT3DBH.EXE`, `STRT3DBL.EXE`, and `STRT3FX.EXE` report DOS/32A exceptions after `Invalid Driver ID`. Under original DOS/4GW, `STRT3DBH.EXE` and `STRT3DBL.EXE` trigger DOSBox `E_Exit: RET from illegal descriptor type 12`, and `STRT3FX.EXE` reports `Fatal error: unable to load DLL` after the same hardware-driver failure.

- Observation: The real-hardware crash is inside DOS/32A's default protected-mode IRQ handler, not the LE loader.
  Evidence: The crash address `0008:00001AD1` maps to `irq_standard`'s final `iretd` in `src/dos32a/text/kernel/intr.asm`; the following bytes are the start of `irq_callback`.

- Observation: S2/SR use a protected-mode IRQ0 timer handler that can chain to the old protected-mode IRQ0 vector.
  Evidence: In `START65H.EXE`, the install routine performs INT 31h AX=0204h/0205h for BL=08h and stores the old CX:EDX vector. The runtime chain path restores the interrupted stack and executes a far return to the old vector.

- Observation: The 65k-only failure is consistent with reentrant timer IRQ exposure while chaining to the default handler.
  Evidence: The game handler EOIs IRQ0 early and executes `sti`; DOS/32A's default handler reflects the IRQ to real mode. The crash frame at DOS/32A's final `iretd` contains an invalid return CS selector (`F053` in the user's photo), which suggests the interrupted stack frame has been corrupted before the default handler returns.

- Observation: The first IRQ-mask candidate had an implementation bug independent of the original S2 crash.
  Evidence: The real-hardware crash moved to `pop es` at `0008:00001B2D`, and listing inspection showed `BX` was no longer holding the prepared real-mode SP when `sub bx,12` ran. That could make the real-mode reflected handler run on an invalid stack.

## Plan of Work

First, classify each game's launch path. Read `S2.BAT`, `SR.BAT`, and `SETUP.BAT` where present. Use SB or local byte inspection from DOSBox-X only when needed, because the built utilities are DOS executables and cannot run directly under 64-bit Windows. Record which EXE each batch launches and whether that EXE is an MZ real-mode launcher, a bound linear executable, or a DOS/4GW-client payload.

Second, reproduce the user's report. Run setup and game launches through DOSBox-X with the current active `DOS4GW.EXE` in each folder. Capture DOSBox-X console logs. The important outcomes are whether setup returns, whether the game crashes before showing game text, whether DOSBox-X logs an `E_Exit`, CPU error, illegal descriptor, or DOS/32A fatal message, and whether the failure differs between S2 and SR.

Third, compare against original DOS/4GW only if needed. Because each folder has `DOS4GW.EXE.bak`, temporarily rename the active replacement to a documented `D32*.EXE` scratch name, copy `.bak` to `DOS4GW.EXE`, run the same launch for up to 10 seconds, then restore the replacement immediately. This step proves whether the game itself can launch in the current DOSBox-X configuration and prevents misclassifying missing sound/video configuration as a DOS/32A bug.

Fourth, analyze approachability. A failure is approachable for 26.1 if it points to a small, existing DOS/32A surface such as a DPMI function return value, interrupt simulation bug, environment/config default, or launch-path detection issue. A failure is not approachable if it requires a large new executable-format loader, substantial DOS/4GW-specific API emulation, or invasive protected-mode memory manager changes.

## Idempotence and Recovery

Do not permanently modify `DOS4GW.EXE`, `DOS4GW.EXE.bak`, game EXEs, configuration files, or data folders. If a test temporarily swaps `DOS4GW.EXE`, restore it in a `finally` block before exiting the host script. Scratch files must begin with `D32` and be removed at the end of each run. If DOSBox-X times out, kill only the process started for the test, then verify no `dosbox-x.exe` processes remain.

If a test crashes or exits halfway through a temporary swap, restore `DOS4GW.EXE` from the saved scratch replacement or `.bak` before continuing. Do not delete user-provided files that do not begin with the documented scratch prefix.

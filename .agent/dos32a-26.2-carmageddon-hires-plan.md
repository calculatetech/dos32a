# Restore Carmageddon High-Resolution Renderer Loading

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `.agent/PLANS.md` in this repository. A future contributor should be able to continue the work from this file alone.

## Purpose / Big Picture

Carmageddon starts through a DOS/4GW-style launcher stub in `MAINPROG.EXE`. The normal game path starts with `CARMA.BAT`; the high-resolution path starts with `HIRES.BAT`, which adds `-hires`. The user reports that replacing DOS/4GW with DOS/32A fails only in high-resolution mode with a driver-loading error. After this work, the high-resolution path should load its renderer module with DOS/32A the same way it does with DOS/4GW, without breaking the 26.1 compatibility fixes for S2/SR, BUILD-engine write combining, or id Tech games.

## Surprises & Discoveries

- Observation: The visible message described as a driver failure appears in `MAINPROG.EXE` as `Failed to load renderer`; the same binary also contains BRender-related strings including `.BDD`, `Default-Renderer`, `softrend`, `softrnd0`, `Default-Primitives`, and `Default-Format`.
  Evidence: `rg -a -n "Failed to load renderer|Default-Renderer|BDD" C:\DOSBox-X\drivec\CARMA\MAINPROG.EXE`.

- Observation: Carmageddon's current directory contains both `DOS4GW.EXE` and `DOS4GW.32`. `DOS4GW.EXE` is the original Rational DOS/4GW runtime. `DOS4GW.32` is a DOS/32A binary, but the Watcom stub in `MAINPROG.EXE` asks for `dos4gw.exe`, so `DOS4GW.32` is not used by the normal launch path unless another script renames or copies it.
  Evidence: `rg -a -n "dos4gw.exe|DOS/4GW|DOS/32A" C:\DOSBox-X\drivec\CARMA\MAINPROG.EXE C:\DOSBox-X\drivec\CARMA\DOS4GW.EXE C:\DOSBox-X\drivec\CARMA\DOS4GW.32`.

- Observation: DOSBox-X does not reproduce the real-hardware failure. Original DOS/4GW and DOS/32A both reach a 640x480 graphical mode and remain running until killed at 10 seconds. This is still useful because a DOSBox-X console failure would have indicated a simple loader regression, but the absence of one means real hardware remains the acceptance environment.
  Evidence: `C:\DOSBox-X\drivez\CARMA-ORIG.LOG`, `C:\DOSBox-X\drivez\CARMA-D32A261.LOG`, and `C:\DOSBox-X\drivez\CARMA-D32A262APIHIDE.LOG` all lack `Failed to load renderer`; the verbose candidate log shows `Loading program "C:\CARMA\MAINPROG.EXE", LE-style` and startup at `CS:EIP=0828:002224F0`.

- Observation: The `.BDD` renderer modules are PE-format BRender modules, not plain data files. They contain import module names such as `BRHOST1`, `BRCORE1`, and `BRPMAP1`, plus exported driver names such as `vesa.dll`, `softrndf.dll`, and `_BrDrv1Begin`.
  Evidence: reading each `.BDD` file's MZ `e_lfanew` value shows PE signatures; string inspection of `VESA.BDD`, `SOFTRNDF.BDD`, and related files shows BRender import and export names.

- Observation: DOS/32A's `RATIONAL DOS/4G` vendor-specific API path currently returns an entry point for an API that is intentionally not implemented. If an application asks DPMI function `0A00h` for `RATIONAL DOS/4G`, DOS/32A returns a far pointer whose code sets `AX=8500h` and jumps to the client critical handler, producing `DOS/4G API calls not supported`.
  Evidence: `src/dos32a/text/kernel/int31h.asm`, labels `int31h_0A00` and `@@ent2`; `src/dos32a/text/client/debug.asm`, `critical_handler`.

- Observation: Startup config order is built-in or embedded defaults first, DOS4GW-name defaults second, and `DOS32A=` environment overrides last. That means environment settings already override the new DOS4GW smart defaults, but configured `.D32` settings needed an explicit stock-config guard to avoid being replaced.
  Evidence: `src/dos32a/dos32a.asm`, calls to `get_default_config`, `apply_dos4gw_name_config`, and `get_environ_config`.

- Observation: Carmageddon runs with the DOS/4G API hidden, but hangs on exit. That makes API hiding a suspect compatibility shortcut rather than a complete DOS/4GW-equivalent behavior.
  Evidence: user real-hardware report after testing staged SHA256 `93982CA615305F8BD8069D413E68C7271AFEA2B0F996F3EF0BAF6CDDF5C56595`.

- Observation: DOS/4G vendor API function 0 is a package/action lookup and returns a far pointer, while zero is the natural "not found" result. Carmageddon high-resolution renderer loading likely needs DOS/32A to avoid crashing when this API is probed, not necessarily to hide the whole DOS/4G vendor API.
  Evidence: `ref\DOS_4G(W) Unofficial Programmer's Guide.htm`, section "DOS/4G API"; `ref\DOS4G .ver.2.60.English\INCLUDE\DOS32LIB.H`.

- Observation: The old `apply_dos4gw_name_config` path made runtime behavior depend on the extender executable basename. That violates the compatibility policy for this branch: launch name must not change DOS/32A behavior.
  Evidence: removed call in `src\dos32a\dos32a.asm`; removed helper and `DOS4GW` name constants from `src\dos32a\text\client\misc.asm` and `src\dos32a\text\client\data.asm`.

- Observation: With the DOS/4G API-entry stub change, SS could show "Ignore DOS/4G API Extension Calls" OFF while Carmageddon still took a working API-probe path. That is misleading because the visible configuration no longer explains runtime behavior.
  Evidence: user report that API OFF still let Carmageddon load and then hang on exit; source had changed `@@ent2` in `src\dos32a\text\kernel\int31h.asm`.

- Observation: Direct kernel testing showed that the visible bit and the runtime branch do match. A DOS/32A image with ID32 kernel misc byte `3Fh` returns a `RATIONAL DOS/4G` vendor entry point from INT 31h function `0A00h`; the same image with byte `BFh` returns carry set and error `8001h` for that vendor string.
  Evidence: the temporary `.agent\dos4g_probe.asm` probe printed `DOS4G_API_PRESENT` for byte `3Fh` and `DOS4G_API_IGNORED` for byte `BFh` when run from `C:\DOSBox-X\drivez` with `set DOS32A=Z:\`.

- Observation: DOSBox-X's drive Z overlay is not a valid place to test SS writes to executable headers. Root-level staged programs execute there, but SS reports `SS/32A fatal: could not configure file "TGT.EXE"` when asked to rewrite a target executable on Z:. The same run still executes the target afterward, proving this is a write-path limitation of the test location rather than a runtime-load failure.
  Evidence: root-level SS probe runs on `C:\DOSBox-X\drivez` consistently printed the fatal configure message and left the target ID32 byte unchanged.

- Observation: Exposing a softened DOS/4G vendor API made the "Ignore DOS/4G API Extension Calls" option too weak for the current investigation. With the soft entry point, Carmageddon no longer changed exit behavior when the option was toggled OFF, so the option stopped providing a useful distinction between "hide DOS/4G API" and "advertise DOS/4G API".
  Evidence: user real-hardware report after staged SHA256 `4C7D6A6F702A841647B756A3A600BF0927BFAF7F6551839FC5103BA03AD9BAFF`.

- Observation: On a writable DOSBox-X C: staging directory, SS correctly writes the embedded ID32 kernel misc byte from `.D32` files. Applying `OFF.D32` changed a copied target from `BFh` to `3Fh`; applying `ON.D32` left a copied target at `BFh`. This means the confusing real-hardware behavior was not caused by SS failing to save the byte.
  Evidence: temporary `C:\DOSBox-X\drivec\TMPSSOFF` and `TMPSSON` runs of `SS.EXE TGT.EXE OFF.D32 /S` and `SS.EXE TGT.EXE ON.D32 /S` exited 0; host-side ID32 byte inspection showed `0x3F` and `0xBF` respectively.

- Observation: Carmageddon high-resolution mode depends on DOS/32A selecting HDPMI when HDPMI is loaded. The removed 26.1 DOS4GW-name profile forced that by writing `00001010b` into the active kernel config at runtime. With filename policy removed, the equivalent explicit settings are DPMI/VCPI detection order (`BFh` to `BEh`) through SS, or `DOS32A=/DPMITST:ON` in the environment.
  Evidence: user real-hardware testing with HDPMI loaded versus unloaded; `git show v26.1:src/dos32a/text/client/misc.asm` shows `set_dos4gw_config` writing `pm32_data+0 = 00001010b`; `src/dos32a/text/client/config.asm` clears bit 0 for `/DPMITST:ON`.

- Observation: SS preserves and writes the active kernel bits needed for this issue. Applying `.D32` configs with kernel misc bytes `BFh`, `BEh`, `3Fh`, and `3Eh` to copied targets through `SS.EXE target config.D32 /S` wrote those exact bytes back to the executable headers.
  Evidence: temporary `C:\DOSBox-X\drivec\TMPSSFLG` run produced `TGTBF.EXE: misc=0xBF`, `TGTBE.EXE: misc=0xBE`, `TGT3F.EXE: misc=0x3F`, and `TGT3E.EXE: misc=0x3E`.

- Observation: The removed 26.1 HDPMI path was stronger than normal DPMI-first detection. It checked INT 2Fh AX=1687h for a 32-bit DPMI host and then required the returned entry segment to contain the `HDPMI` signature before accepting the host. Current normal DPMI-first logic lacked that signature-specific shortcut, which matches the user's report that `DPMITST=ON` did not restore Carmageddon or BUILD write combining.
  Evidence: `git show f85ed33:src/dos32a/text/kernel/detect.asm` shows `@@detect_HDPMI`; current `src/dos32a/text/kernel/detect.asm` had only `@@detect_DPMI` and `@@detect_VCPI` before this repair.

- Observation: The apparent `DPMITST=ON` regression was not a valid HDPMI-path test because HDPMI32 was not loaded. The candidate with an explicit HDPMI signature probe was therefore rolled back before further source work.
  Evidence: user report on 2026-05-27; current `src/dos32a/text/kernel/detect.asm` has no `@@detect_HDPMI` diff.

- Observation: 26.1 and 26.1.1 intentionally differ on filename policy. Tag `v26.1` calls `apply_dos4gw_name_config` and compares the extender basename with `DOS4GW`/`DOS4GW.EXE`. The 26.1.1 candidate removes that call, helper, and constants; configuration depends on embedded SS settings and `DOS32A=` environment flags only.
  Evidence: `git show v26.1:src/dos32a/dos32a.asm`; `git show v26.1:src/dos32a/text/client/misc.asm`; `rg apply_dos4gw_name_config|set_dos4gw_config|_dos4gw_name src` has no matches in the current tree.

## Decision Log

- Decision: Treat the problem as a DOS/4GW compatibility issue in the external-loader and dynamic-module path, not as a game-specific batch-file issue.
  Rationale: The failure is reached only when using DOS/32A as the DOS/4GW replacement, and the high-resolution path appears to trigger BRender renderer module loading through `.BDD` files. DOS/32A is marketed as a DOS/4GW replacement, so the first goal is to match DOS/4GW behavior for this class of loader operation.
  Date/Author: 2026-05-26 / Codex.

- Decision: Keep all DOSBox-X iteration runs to 10 seconds or less unless the user explicitly asks for longer.
  Rationale: Prior validation showed that longer waits waste time for this class of launch failure; successful or failed startup behavior appears quickly.
  Date/Author: 2026-05-26 / Codex.

- Decision: Test a candidate that hides the unsupported DOS/4G vendor API by default when DOS/32A is launched through the `DOS4GW.EXE` replacement path.
  Rationale: Returning a successful `RATIONAL DOS/4G` API entry point but then failing every use of that entry point is not faithful DOS/4GW compatibility. DOS/32A already documents the "Ignore DOS/4G API Extension Calls" behavior as a way to force applications away from DOS/4G-only extensions. The candidate applies that documented behavior to the replacement-name defaults, which is general and not Carmageddon-specific.
  Date/Author: 2026-05-26 / Codex.

- Decision: Preserve explicit SS/embedded `.D32` settings ahead of the DOS4GW-name smart defaults.
  Rationale: The smart default is meant to make an unconfigured `DOS4GW.EXE` replacement behave better, not to disable documented configuration knobs. If the runtime image has been configured, those settings should be treated as deliberate. `DOS32A=` environment overrides remain last and still win.
  Date/Author: 2026-05-26 / Codex.

- Decision: Do not force "Ignore DOS/4G API Extension Calls" for the DOS4GW replacement default.
  Rationale: It fixes Carmageddon startup but regresses exit behavior. A more compatible default is to advertise the DOS/4G vendor API as DOS/4GW does, while making unsupported API calls fail benignly instead of raising DOS/32A's critical `DOS/4G API calls not supported` handler.
  Date/Author: 2026-05-26 / Codex.

- Decision: Remove all filename-based extender policy.
  Rationale: DOS/32A must behave the same when invoked as `DOS32A.EXE`, `DOS4GW.EXE`, or any other name. Configuration must come from the embedded/SS `.D32` settings or documented environment options only. This is easier to explain, test, and trust than hidden compatibility behavior tied to launch circumstances.
  Date/Author: 2026-05-26 / Codex.

- Decision: Make "Ignore DOS/4G API Extension Calls" ON by default and keep the original DOS/4G unsupported-API entry behavior.
  Rationale: The user verified that Carmageddon works when that documented option is ON. Changing the API implementation made behavior diverge from what SS reported. The safer release behavior is to expose the compatibility choice through the existing configuration bit and make the shipped default match it.
  Date/Author: 2026-05-26 / Codex.

- Decision: Revert the softened DOS/4G API entry behavior and keep the API implementation historical for this release point.
  Rationale: The softened entry may be a valid future compatibility direction, but it made the current ON/OFF option behaviorally indistinguishable for Carmageddon and obscured whether the documented setting was truly being honored. For 26.2, the clearer state is: ON hides the DOS/4G vendor API, OFF advertises the historical unsupported API entry.
  Date/Author: 2026-05-26 / Codex.

- Decision: Do not make DPMI-first the global default as part of this fix.
  Rationale: The DPMITST environment/config flag already exists to select DPMI/VCPI order. Carmageddon should be fixed by proving and using that explicit setting, not by adding a new global default or restoring hidden filename-dependent policy.
  Date/Author: 2026-05-27 / Codex.

- Decision: Restore the HDPMI signature probe, but make it conditional on explicit DPMI-first configuration instead of filename or game detection.
  Rationale: The user wants documented configuration flags to control behavior and no special treatment for `DOS4GW.EXE`. Tying the HDPMI probe to `pm32_mode` bit 0 means SS DPMI-first and `DOS32A=/DPMITST:ON` select the old HDPMI-preference path, while the default VCPI-first path remains unchanged.
  Date/Author: 2026-05-27 / Codex.

- Decision: Roll back the HDPMI signature-probe candidate and avoid further DOSBox drive staging unless the issue reproduces in DOSBox.
  Rationale: The failing test was missing HDPMI32, so the source hypothesis was not proven. The user only needs binaries in `binw` for real-hardware testing, and persistent DOSBox drive edits add noise when DOSBox cannot reproduce the failure.
  Date/Author: 2026-05-27 / Codex.

- Decision: Ship the current corrected behavior as 26.1.1 rather than 26.2.

## Outcomes & Retrospective

Current implementation outcome: DOS/32A no longer applies any runtime policy based on the extender executable name. Configuration comes from embedded/SS `.D32` settings and `DOS32A=` environment options only. The embedded and SS restore-default kernel misc byte is `BFh`, so VCPI/DPMI detection order and "Ignore DOS/4G API Extension Calls" ON remain the default visible state. DPMI-first operation is selected explicitly with SS (`BFh` to `BEh`) or with `DOS32A=/DPMITST:ON`. When bit 7 is ON, DOS/32A hides the `RATIONAL DOS/4G` vendor API. When bit 7 is OFF, DOS/32A exposes the historical unsupported DOS/4G API entry. Real-hardware testing confirmed Carmageddon high-resolution launch, SS toggles, DPMITST behavior, and BUILD LFB write-combining with HDPMI32 loaded. The release target is now 26.1.1.

## Context and Orientation

The repository root is `C:\Users\mbeutler\Projects\dos32a`. The active branch for this work is `26.2`. Build outputs are written to `binw\`. The DOSBox-X test games live outside the repository under `C:\DOSBox-X\drivec`.

DOS/4GW is the protected-mode runtime originally used by many Watcom C/C++ DOS games. A small real-mode stub at the front of a game executable locates and runs `DOS4GW.EXE`, then the extender loads the protected-mode game. DOS/32A can be used as a replacement when it is copied or renamed to the external loader name the game expects, often `DOS4GW.EXE`.

Carmageddon files relevant to this plan:

- `C:\DOSBox-X\drivec\CARMA\HIRES.BAT` prints a high-resolution banner and calls `Carma.BAT -hires`.
- `C:\DOSBox-X\drivec\CARMA\CARMA.BAT` runs `MAINPROG.EXE -runme` and forwards all arguments.
- `C:\DOSBox-X\drivec\CARMA\MAINPROG.EXE` is the Watcom application executable. It contains the DOS/4GW stub string `dos4gw.exe` and high-resolution renderer-loading strings.
- `C:\DOSBox-X\drivec\CARMA\*.BDD` are BRender driver modules. The high-resolution mode likely changes which renderer module is selected.
- `C:\DOSBox-X\drivec\CARMA\DOS4GW.EXE` is currently the original Rational DOS/4GW runtime.
- `C:\DOSBox-X\drivec\CARMA\DOS4GW.32` is a DOS/32A binary but is not the launcher filename requested by `MAINPROG.EXE`.

Repository files likely relevant to a fix:

- `src/dos32a/` contains the DOS/32A extender.
- `src/dos32a/text/kernel/int31h.asm` implements DPMI interrupt `31h` services.
- `src/dos32a/text/client/` contains the client-side startup and DOS/4GW compatibility logic.

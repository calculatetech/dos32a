# Investigate Death Rally PMODE/W compatibility for DOS/32A 26.1

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `.agent/PLANS.md` from the repository root. A future contributor should be able to restart from this file alone, with the current working tree and the local DOSBox-X test material.

## Purpose / Big Picture

DOS/32A is marketed as a compatible replacement for PMODE/W, a compact DOS protected-mode extender used by some games. Death Rally is a real game in `C:\DOSBox-X\drivec\RALLY` whose `RALLY.EXE` is currently PMODE/W-bound and is reported to crash hard when rebound to DOS/32A. The goal for DOS/32A 26.1 is not a broad rewrite; it is to find low-hanging compatibility defects that can be reproduced, fixed narrowly, and validated without destabilizing working DOS/32A behavior.

After this work, a maintainer should know whether Death Rally exposes a small DOS/32A PMODE/W compatibility bug suitable for 26.1. If it does, the branch should contain the fix, a clean build, a reproducible DOSBox-X validation transcript, and updated notes. If it does not, the branch should contain a clear investigation record explaining why the issue is deferred.

## Surprises & Discoveries

- Observation: Death Rally's game directory does not currently include an original `.BAK` beside `RALLY.EXE`, despite other game folders being described as having originals.
  Evidence: Listing `C:\DOSBox-X\drivec\RALLY` showed `RALLY.EXE`, `SETUP.EXE`, `DR.CFG`, and data archives, but no `.BAK`.

- Observation: The existing source already contains a PMODE/W compatibility DPMI call.
  Evidence: `src/dos32a/text/kernel/int31h.asm` contains a dispatch entry for `0EEFFh` and a section labeled `PMODE/W id function`.

- Observation: Death Rally's protected-mode payload starts with `PMW1`, not LE, LX, LC, or PE.
  Evidence: The MZ header has `e_lfanew=0x2800`; bytes at `0x2800` are `PMW1`; the PMW1 header reports version 1.24, compression flag `0x0001`, four objects, object table offset `0x28`, relocation table offset `0x88`, and data pages offset `0x1A25F`.

- Observation: The DOS/32A loader does not implement PMW1. It searches a bound or external application for LE, LX, LC, or PE signatures before dispatching to format-specific loaders.
  Evidence: `src/dos32a/dos32a.asm` dispatches only to `load_le_app`, `load_lx_app`, `load_lc_app`, and `load_pe_app` after reading the application signature.

- Observation: PMW1 support is a separate loader feature, not a one-line signature alias.
  Evidence: The public PMODE/W source describes a PMW1 header, object table, relocation table entries, and decompression path. Death Rally sets the PMW1 compression flag, so a compatible loader would need to allocate objects, decompress object data, decompress relocation blocks, and apply PMW1 relocations.

## Decision Log

- Decision: Treat Death Rally as a PMODE/W compatibility investigation, not as part of the previous SB chained MZ/LE binding bug.
  Rationale: `SB.EXE` reports `Application Type: PMODE/W DOS Extender bound to`, and preliminary marker inspection found PMODE/W markers without a chained LE header matching the previous issue.
  Date/Author: 2026-05-23 / Codex

  Rationale: No `.BAK` original is visible, and the user-provided game tree is test material rather than repository source. Tests must create temporary bound copies and remove them afterward.
  Date/Author: 2026-05-23 / Codex

- Decision: Run DOSBox-X tests with an explicit `mount c C:\DOSBox-X\drivec`, `c:`, and a 10-second cap.
  Rationale: The user reported DOSBox-X is not auto-mounting drive C, and also stated no iteration test needs more than 10 seconds.
  Date/Author: 2026-05-23 / Codex

- Decision: For 26.1, fix SB/32A to refuse PMW1 bind requests instead of adding a PMW1 loader to DOS/32A.
  Rationale: The user asked to keep 26.1 focused on low-hanging compatibility fruit. Full PMW1 support requires new format loading, decompression, and relocation behavior. SB already knows when a payload is PMW1, so refusing to create a DOS/32A-bound executable that the runtime cannot load is the smallest corrective change and prevents a misleading replacement workflow.
  Date/Author: 2026-05-23 / Codex

## Outcomes & Retrospective

Interim outcome: Death Rally is confirmed to be a compressed PMW1 PMODE/W executable. DOS/32A 26.1 does not gain full PMW1 execution support in this plan. Instead, SB/32A now makes the unsupported case explicit and stops before producing a broken replacement executable. This turns a confusing runtime failure into a bind-time diagnostic and keeps a future PMW1 loader as a separate, larger compatibility project.

Validation outcome: the full repository build completed; Death Rally now prints `SB/32A fatal: cannot bind PMW1-compressed PMODE/W application "C:\RALLY\RALLY.EXE"` and no temporary output executable is produced; a DOOM2 DOS/4G regression bind still produces a DOS/32A-bound LE executable; the existing DOSBox-X smoke test passed.

## Context and Orientation

The repository root is `C:\Users\mbeutler\Projects\dos32a`. Source for the DOS/32A extender is under `src/dos32a`; the bind utility SB, which replaces DOS extender stubs, is under `src/sb`. Built Windows-hosted utilities and DOS binaries are under `binw` after `build.cmd` succeeds.

PMODE/W is a DOS protected-mode extender. A DOS extender switches a DOS program from real mode into protected mode so the program can use 32-bit code and more memory. DOS/32A claims compatibility with PMODE/W in the bundled 8.00 documentation. In this repository, PMODE/W compatibility appears in two places. The bind utility in `src/sb/sbind.asm` can detect a PMODE/W-bound executable. The runtime extender in `src/dos32a/text/kernel/int31h.asm` implements DPMI function `0EEFFh`, named in the bundled docs as "Get DOS Extender Info (PMODE/W compatible)". DPMI means DOS Protected Mode Interface, the interrupt API used by protected-mode DOS programs to ask the extender for memory, selectors, interrupts, and extender metadata.

The local test game is `C:\DOSBox-X\drivec\RALLY`. It contains `RALLY.EXE` plus data files such as `MENU.BPA`, `MUSICS.BPA`, and `SANIM.HAF`. This plan assumes DOSBox-X is installed at `C:\DOSBox-X` and must be invoked with an explicit mount for `C:\DOSBox-X\drivec`.

## Plan of Work

First, inspect the PMODE/W compatibility implementation. Read `src/dos32a/text/kernel/int31h.asm` around the DPMI dispatch table and `0EEFFh` handler, then compare the same areas in `dos32a_800/src/dos32a/text/kernel/int31h.asm` and `dos32a-735-src/src/dos32a/text/kernel/int31h.asm`. Read `dos32a_800/docs/html/prog/int31h/0eeff.htm` to capture the expected register contract. Also inspect `dos32a_800/src/sutils/pctest/pctest.asm`, because it contains known detection code for PMODE/W-compatible extenders.

Second, reproduce the failure without modifying the original game executable. Copy the current built `SB.EXE` and required DOS/32A binary into a temporary location under `C:\DOSBox-X\drivec`, bind `RALLY.EXE` to a temporary executable name such as `D32RALLY.EXE`, and run that temporary executable from DOSBox-X. Capture DOSBox-X stdout/stderr and any visible crash signature. Kill DOSBox-X after 10 seconds if it is still running. This reproduction has shown that the old SB behavior generated a temporary DOS/32A-bound executable that then failed with DOS/32A fatal 3004 because the payload was PMW1.

Third, use the reproduction to decide whether the issue is low-hanging. Low-hanging means a small, explainable change in an existing compatibility path, such as a wrong return value from `0EEFFh`, a missing PMODE/W configuration default, or an SB replacement behavior that can be fixed without rewriting loader architecture. If the issue instead requires a broad loader rewrite, unknown PMODE/W binary format support, or invasive protected-mode changes, document it and defer it.

Fourth, if a narrow fix is identified, implement it in the smallest source area that owns the behavior. Preserve legacy formatting and license headers. Rebuild with `cmd /c build.cmd`. Validate with Death Rally and with existing smoke/regression checks. Update this plan, `ChangeLog` if user-visible behavior changes, and any relevant documentation. The chosen fix is in `src/sb/main.c`, where bind requests now abort when `Main_Type==5` or `Exec_Type==5`, the internal values SB uses for PMW1 payloads.

## Interfaces and Dependencies

The primary interface under investigation is DPMI interrupt `INT 31h`, function `AX=0EEFFh`. In DOS/32A this is implemented in `src/dos32a/text/kernel/int31h.asm`. If changed, the handler must keep the dispatch table entry for `0EEFFh`, preserve carry-flag error behavior for unsupported or invalid functions, and return values consistent with the bundled DOS/32A documentation or with PMODE/W compatibility evidence from the included PCTEST source.

The secondary interface is SB binding for PMODE/W executables in `src/sb/sbind.asm`. If changed, SB must continue to detect PMODE/W-bound files and must not regress the already-fixed DOS/4G BW/MZ/LE bind path.

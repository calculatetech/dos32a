# Fix SB/32A BW Overlay Binding for 26.1

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `.agent/PLANS.md` from the repository root. It is self-contained so a future contributor can continue from the current `26.1` branch, the source tree, and the test material staged under `C:\DOSBox-X\drivec`.

## Purpose / Big Picture

Version 26.0 is the first stable release, but an old `sb.exe` report says DOS/32A's bind utility cannot replace the stub for IndyCar Racing II because the executable has an incompatible first `MZ` section and a second `BW` section. After this work, the 26.1 branch should prove whether that bug still exists in the active source, fix it if it does, and validate that `sb.exe` can replace stubs for the in-scope game set while keeping unrelated failures out of this release.

The main observable result is that `SB.EXE` can inspect, unbind, and bind the IndyCar Racing II executable from `C:\DOSBox-X\drivec\ICR2` without rejecting it solely because the second section is `BW`. Runtime validation should use DOSBox-X with `C:\DOSBox-X\drivec` as drive `C:` and stop each run after no more than 10 seconds.

## Surprises & Discoveries

- Observation: The active source already accepts `BW` while walking chained executable sections.
  Evidence: `src\sb\sbind.asm` contains `cmp wptr [edx],'WB'`, which is the little-endian source literal for a file signature stored as `BW`. The same check exists in the 7.35 and 8.00 reference trees.
- Observation: The related SR loader walks chained headers by accepting both `MZ` and `BW` signatures.
  Evidence: The referenced source checks for `MZ`, subtracts 512 from the computed size for `MZ`, and otherwise accepts `BW`; any other signature is treated as invalid before moving to the next header.
- Observation: The active failure is caused by missing the valid embedded `LE` header, not by failure to skip `BW`.
  Evidence: ICR2 `INDYCAR.EXE` has section 0 `MZ` at `0x00000000`, section 1 `BW` at `0x0000F474`, and section 2 `MZ` at `0x00026654` with `e_lfanew=0x2648`, so the true `LE` is at `0x00028C9C`. Before the fix, `SB.EXE /U` output `OUT.LC` size `641816`, which starts at a later false `LC` signature and contains LC version bits `9`; the DOS/32A LC loader supports version `4`, so the rebound output failed with fatal 3006.
- Observation: The patched `SB.EXE` extracts the correct linear executable from ICR2.
  Evidence: After the fix, `SB.EXE /U /UNOUT.LE INDYCAR.EXE` outputs `OUT.LE` size `975319`, and inspecting `OUT.LE` reports `Unbound LE-style file format Linear Executable`.

## Outcomes & Retrospective

The branch now contains a targeted `SB/32A` fix for the ICR2 chained-header case. The old report is not quite accurate for this tree: `BW` support was present, but the chained MZ after the BW section was treated only as a size record, so `SB.EXE` skipped past the real `LE` header and later matched unrelated bytes that looked like `LC`. The fix makes `SB.EXE` use the embedded MZ `e_lfanew` pointer and produce a correct LE output.

Validation passed for `cmd /c build.cmd`, `cmd /c "smoke-dosbox.cmd -TimeoutSeconds 10"`, copied ICR2 unbind/rebind, temporary ICR2 launch as `INDY32.EXE`, and a copied DOOM2 unbind/rebind regression.

## Context and Orientation

`src\sb` contains the active SUNSYS Bind Utility. `src\sb\main.c` handles command-line flow and messages. `src\sb\sbind.asm` contains the low-level executable detection, unbinding, and binding code that reads executable headers. A DOS `MZ` header is the normal 16-bit executable header. `LE`, `LX`, and `LC` are linear-executable formats used by DOS extenders. `BW` is the signature used by Watcom-style executable sections seen in the IndyCar Racing II report.

`C:\DOSBox-X\drivec` contains game directories. In-scope folders are the games not explicitly reserved for later and not identified as PMODE-only. The user explicitly reserved `CARMA`, `CARSPLAT`, `S2`, and `SR` for later. `RALLY` contains Death Rally and is PMODE; inspect it only enough to decide whether it is connected to the bind bug.

## Plan of Work

First, inspect `src\sb\sbind.asm` around the file-type detection and chained-header walking logic. The fix should be as small as possible: recognize when a chained `MZ` header is itself a valid linear-executable stub by checking relocation table offset `0040h` and nonzero `e_lfanew`, then set `_exec_start` to the absolute `MZ+e_lfanew` address and `_app_off_datapages` to the embedded MZ base. Preserve existing behavior for `MZ`, `BW`, `LE`, `LX`, and `LC`.

Next, build `sb.exe` with `cmd /c build.cmd sb`. Test on copies of the user's `.bak` files in a scratch directory under the repository or under a temporary `C:\DOSBox-X\drivec\D32*` directory. Do not overwrite the user's current game executables during bind/unbind tests unless explicitly restoring from `.bak` in the same script.

Then, validate runtime with DOSBox-X. Each DOSBox-X run must be capped at 10 seconds or less, should log console output, should be killed in a `finally` block if still running, and must delete temporary batch/log files. If a game reaches a menu, intro, setup error, or known non-crash marker, record that as the acceptance signature for that game.

# Restore DOS4GW Replacement LFB Write Combining Without Regressing id Tech

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `.agent/PLANS.md` from the repository root.

## Purpose / Big Picture

The user reports that linear frame buffer write combining is broken again in DOS4GW.EXE replacement use after the 26.1 S2/SR work. Write combining is the CPU or DPMI-host behavior that lets linear frame buffer writes be combined for much faster video output; on this hardware it previously produced about a 20 FPS Redneck Rampage improvement. The goal is to restore that DOS4GW replacement behavior without reverting the id Tech HDPMI32 fix that keeps Doom-family bound executables from freezing when HDPMI32 is loaded.

The observable result is a rebuilt `binw\dos32a.exe` that still contains the S2 selector-pair fix, still avoids forced HDPMI32 for Doom-family bound executables, and again tries the HDPMI32 external-DPMI path when the extender itself is launched as `DOS4GW.EXE`. This is a compatibility-class rule rather than a game-name list: any game that loads an external `DOS4GW.EXE` gets the DOS4GW replacement policy, while bound executables such as the tested Doom-family binaries remain on the non-forced path.

## Surprises & Discoveries

- Observation: The regression is probably not caused directly by the S2 selector-pair allocation.
  Evidence: The only source change since the known `b77e519` id Tech fix in the HDPMI detection area is absent; the selector-pair commit only touched `src/dos32a/loader.asm`. The WC behavior instead depends on the earlier `detect.asm` decision to stop forcing HDPMI32 before the IOPL/XMS shortcut.

- Observation: The user's local broad game patch package predates the id Tech HDPMI fix.
  Evidence: `dist\dos32a-26.1-drivec-patch\MANIFEST.TXT` records source commit `c54df59`, while the id Tech fix is commit `b77e519`. That explains how BUILD WC could still appear to work while the source tree had already moved away from forced HDPMI32.

- Observation: The name-table matcher has to stay within 8086 addressing modes because it lives before the `.386p` section of `misc.asm`.
  Evidence: The first build failed with `TEXT\CLIENT\misc.asm(226) Illegal indexing mode` for `mov bl,[dx]`. Rewriting the table pointer to use `BX` fixed the build.

- Observation: DOS/32A's `/DPMITST:ON` does not mean "prefer DPMI"; it means "try VCPI before DPMI."
  Evidence: `src\dos32a\text\client\config.asm` `cfg_env_test` sets bit 0 of `pm32_data[0]` when the switch is on, and `src\dos32a\text\kernel\detect.asm` treats that bit as `VCPI/DPMI` order.

- Observation: DOS/32A's internal VCPI physical mapper intentionally disables page caching for mapped device pages.
  Evidence: `src\dos32a\text\kernel\int31h.asm` function `int31h_0800` uses page attribute `1Fh` on 486+ CPUs, with the local comment "set PCD and PWT bits (no page cache)." This matches the DPMI note in `dos32a_800\docs\html\prog\int31h\0800.htm`, but it means VCPI mode should not be expected to provide LFB write combining.

## Decision Log

- Decision: Do not revert the id Tech fix.
  Rationale: Restoring unconditional HDPMI32 detection would likely bring back the Doom-family freezes that `b77e519` fixed. The replacement must be narrower than the original 26.0 HDPMI shortcut.
  Date/Author: 2026-05-24 / Codex

- Decision: Force early HDPMI32 only for known BUILD-engine executable names.
  Rationale: The BUILD performance benefit depends on HDPMI32 being selected as the DPMI host before DOS/32A takes the XMS/IOPL shortcut. id Tech needs the opposite behavior. At this point the practical low-risk split is by launch target: known BUILD targets get the HDPMI32 path, while Doom-family executables continue using the non-forced detection path.
  Date/Author: 2026-05-24 / Codex

- Decision: Replace the BUILD executable-name list with a DOS4GW.EXE replacement policy.
  Rationale: The user needs a compatibility argument that can be defended to DOS users, not a list of whatever games happened to be tested. The common compatibility boundary in the current drive C test set is whether DOS/32A is being loaded as an external `DOS4GW.EXE`. That boundary covers Redneck Rampage, Screamer 2, Screamer Rally, and other external-DOS4GW games without touching bound id Tech binaries.

## Context and Orientation

`src\dos32a\text\kernel\detect.asm` decides whether DOS/32A runs under an external DPMI host such as HDPMI32, under VCPI, or using its XMS/raw path. A DPMI host is software that provides protected-mode services to DOS programs. HDPMI32 can apply write combining to physical linear frame buffer mappings, which is why BUILD games run faster when DOS/32A chooses HDPMI32.

Commit `b77e519` removed an HDPMI32-specific early-detection shortcut because forcing HDPMI32 caused id Tech games to freeze on the user's real hardware. The current regression report indicates external-DOS4GW games need a way back to that early HDPMI32 path, while bound id Tech games must not be forced there until the external-DPMI id Tech freeze is understood.

`src\dos32a\text\client\misc.asm` runs before `pm32_info` and already contains `apply_dos4gw_name_config`, which recognizes when the extender itself was launched as `DOS4GW.EXE` and installs the archived `DOS4GW.D32` defaults. That makes it the right place to set the early-HDPMI preference for DOS4GW replacement mode. `src\dos32a\kernel.asm` owns the kernel-side variables read by `detect.asm`.

## Plan of Work

Use the existing kernel byte, `pm32_hdpmi`, that the real-mode client can set before calling `pm32_info`. Bit 0 means "try HDPMI32 before the normal detection order and before the IOPL/XMS shortcut." Keep the default at zero so Doom-family bound executables remain on the fixed path.

In `src\dos32a\text\client\misc.asm`, make `set_dos4gw_config` call `set_hdpmi_force` after installing the DOS4GW.D32 defaults. Remove the separate BUILD-engine executable-name table and matcher. Do not inspect the game executable name or command-line token for this decision.

In `src\dos32a\text\kernel\detect.asm`, restore the small `@@detect_HDPMI` helper from the earlier 26.0 path, but call it only when `pm32_hdpmi` bit 0 is set. The calls should happen before the XMS IOPL shortcut and before the normal VCPI/DPMI order in the no-XMS branch.

Update `ChangeLog` to record that 26.1 restores HDPMI32 write-combining for DOS4GW.EXE replacement use without game-name matching.

## Validation and Acceptance

Local validation can only prove that the extender still builds and the generic DOSBox-X smoke path does not crash. Real acceptance requires the user's hardware:

1. With HDPMI32 loaded through the existing `TWEAKS.BAT`, an external-DOS4GW game such as Redneck Rampage should regain the write-combining performance improvement when its `DOS4GW.EXE` is this candidate.
2. Doom-family games should not return to the HDPMI32 launch freeze fixed by `b77e519`.
3. S2 `START65H.EXE` should still pass the user's real-hardware 65k-color test because the selector-pair change remains in place.

## Idempotence and Recovery

The package copy step is safe to repeat because it preserves a `DOS4GW.BAK` when updating a game directory through `UPDATEGW.BAT`. The local drive C staging step intentionally overwrote the listed external `DOS4GW.EXE` test copies with this candidate.

The new kernel variable is:

    pm32_hdpmi db 0

`src\dos32a\dos32a.asm` must declare it as an external byte so client code can write it after loading `DS` with `_seg_kernel`. `src\dos32a\text\kernel\detect.asm` must read it directly in the kernel segment.

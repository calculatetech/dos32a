This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The 26.1.1 branch is stable in real-hardware compatibility testing, but a clean-context adversarial review found correctness and maintenance issues that should be fixed before treating the tree as a trusted base for the next compatibility release. After this work, SB will not destroy an input executable when binding fails, STUB32C will ship the same default DOS/4G API behavior as DOS32A and SS, DOS-facing filenames and paths will be consistently uppercase, version reporting will have one documented convention, and the repository will require clean-context adversarial review for every future change. The result is visible by rebuilding the binaries, inspecting the ID32 defaults in `BINW\DOS32A.EXE` and `BINW\STUB32C.EXE`, running the smoke test, and checking that failure tests preserve their input files.

## Surprises & Discoveries

- Observation: The current tree has no `.gitattributes`, and `git ls-files --eol` reports many files with LF in the index but CRLF or mixed line endings in the working tree. This explains the repeated risk of bloated line-ending diffs.
  Evidence: `git ls-files --eol` shows examples such as `AGENTS.md` as `i/lf w/crlf`, `build.cmd` as `i/lf w/mixed`, and multiple active `src/` files as `i/mixed w/mixed`.

- Observation: The clean-context review found that `BINW\DOS32A.EXE` and `BINW\STUB32C.EXE` do not carry the same ID32 kernel misc default.
  Evidence: `Format-Hex -Path BINW\DOS32A.EXE -Count 128` shows `ID32 BF`; `Format-Hex -Path BINW\STUB32C.EXE -Count 128` shows `ID32 3F`.

- Observation: A full renormalization dry run was too broad for this correctness change.
  Evidence: `git add --renormalize .` touched active sources, reference trees, and bundled historical tools, so the work was reset before staging and `.gitattributes` now preserves historical working-tree line endings until a dedicated mechanical cleanup.

- Observation: DOSBox-X's internal `Z:` drive is not a reliable target for the SB failure-preservation test because temporary file creation can fail there.
  Evidence: The validation used a disposable synthetic LC payload staged under `C:\DOSBox-X\drivez` instead.

- Observation: The final clean-context review found that the first implementation over-applied DOS uppercase casing to host-side repository paths.
  Evidence: `dos32a_800` and `tasm5` are tracked with lowercase directory names, so build scripts and documentation now use those real host-side paths while keeping generated DOS-visible names such as `BINW\DOS32A.EXE` uppercase.

- Observation: The smoke wrapper did not forward command-line options to the PowerShell runner.
  Evidence: `smoke-dosbox.cmd -TimeoutSeconds 10` previously used the PowerShell script default; the wrapper now appends `%*` and the timeout override is part of final validation.

- Observation: The final-final clean-context review found that SB's successful-bind publish path still trusted a void `copy_file` helper.
  Evidence: `copy_file` now returns success or failure, reports destination write failures, preserves the source payload until destination publication succeeds, and leaves the temp output available on publish failure.

- Observation: A later clean-context review found the assembler temp-writer could still treat short DOS writes as success.
  Evidence: `bind_exec_` and `unbind_exec_` now compare the returned byte count against the requested write count after `INT 21h AH=40h`; the C fallback copy path also checks `ferror(src)` before accepting EOF as a complete copy.

- Observation: The same-path SB cases were another destructive edge. Asking SB to publish over the same input path could unlink or overwrite the user's source as part of normal cleanup.
  Evidence: `BindExec` and `UnbindExec` now reject equal source and destination paths with a fatal same-file error before opening the publish path.

- Observation: The DOS read calls in `sbind.asm` had the same short-count problem as the DOS write calls.
  Evidence: The stub and executable read loops now compare `EAX` with the requested byte count and branch to the existing error paths on short reads.

- Observation: Functional edits to assembler files can accidentally introduce mixed line endings even with `.gitattributes` in place.
  Evidence: `src/sb/sbind.asm` and `src/stub32a/stub32c.asm` were restored to CRLF after the latest edits, and `.gitattributes` permits CR-at-EOL whitespace for `*.asm`.

- Observation: Textual same-file comparisons were not strong enough for DOS path aliases.
  Evidence: `src/sb/main.c` now canonicalizes both names with Open Watcom `_fullpath`, folds slashes and case, and strips non-root trailing backslashes before comparing. A DOSBox-X test using `/U /UN.\PAYLOAD.LE PAYLOAD.LE` rejected the aliased destination, preserved `PAYLOAD.LE`, and left no `$$SB32$$.TMP`.

- Observation: The assembler path also needed to treat temp-file close/flush errors as write failures.
  Evidence: `bind_exec_` now branches to `@@err5` and `unbind_exec_` branches to `@@err4` when `INT 21h AH=3Eh` closes the temp file with carry set.

- Observation: Generated or fallback utility filenames were still using lowercase extensions in several places.
  Evidence: SB, SC, SS, and SVER now append and display uppercase fallback DOS extensions such as `.EXE`, `.LE`, `.LC`, `.D32`, and `.PMW`. A focused search for lowercase generated DOS extensions in those utilities now returns no matches.

- Observation: Review found that the first backup/restore publish helper recognized `\` and drive separators but not `/`, while canonical same-name checks already normalized `/` to `\`.
  Evidence: `MakeBackupName` in both SB and SC now normalizes `/` to `\` before selecting the backup directory, so `DIR/APP.EXE` and `DIR\APP.EXE` produce the same-directory backup path.

- Observation: The final clean-context review reported no high or medium findings. Its only actionable low was a historical DOS-visible filename in `ChangeLog` that still used lowercase.
  Evidence: `ChangeLog` now writes `DOS32A.LNK`.

## Context and Orientation

This repository contains the DOS/32 Advanced source tree. DOS/32A is a DOS extender: it lets 32-bit protected-mode programs run from DOS. STUB32C is a small configurable stub executable that can be bound to an application and then hands configuration to DOS32A. SB is the bind utility that combines a stub or extender with an application executable. SS is the setup utility that edits DOS32A or STUB32C configuration stored in an `ID32` block inside the executable. SVER is the version utility that reads and reports DOS32A metadata.

The active source paths are host-side repository paths and currently use the historical casing already present in the tree. The DOS-visible artifacts are files and paths that DOS users see or execute, such as `BINW\DOS32A.EXE`, `BINW\STUB32C.EXE`, `C:\DOS32A\BINW`, and commands printed in batch files or documentation. DOS-visible names must be uppercase after this work. Host-side repository and tool paths, such as `dos32a_800`, `tasm5`, and `C:\DOSBox-X\drivez`, keep their actual filesystem casing.

The reviewed defects are:

SB can delete the source payload before checking whether bind succeeded. In `src/sb/main.c`, `BindExec` calls `bind_exec`, closes handles, unconditionally calls `unlink(newname)`, and only then switches on the return code. If `newname` is the user's unbound input such as `GAME.LE`, a failed bind can destroy that input.

STUB32C's embedded ID32 kernel misc byte is still `3Fh`, where bit 7 is off. DOS32A and SS now default to `BFh`, where bit 7 is on and DOS/4G API extension calls are ignored by default. Because STUB32C answers `INT 21h AX=0FF87h` with its own ID32 block and DOS32A copies that block, STUB32C-bound programs can start with old defaults even when DOS32A and SS agree on the new defaults.

`smoke-dosbox.cmd` still probes lowercase `binw\sver.exe` and `binw\dos32a.exe`, even though `build.cmd` now emits uppercase files. The README also documents a smoke test and then says there are no automated tests, which is stale.

Version display currently works for `26.1.1`, but the runtime banner always uses a three-part string while SS and SVER conditionally use two-part or three-part formatting. The implementation should either document the decimal low-byte convention clearly or adjust the banner so future versions cannot diverge.

The repository has no `.gitattributes`, so Git relies on local settings such as `core.autocrlf=true`. On this machine that causes LF-indexed files to appear as CRLF or mixed in the working tree. Future changes should declare expected text and binary handling explicitly.

## Plan of Work

First, update the repository rules. Edit `AGENTS.md` to add a required review workflow: every code, documentation, build-script, binary, or release-artifact change must end with a clean-context adversarial review over the active codebase, and the maintainer must verify the findings locally before commit. Also add naming guidance that DOS-visible filenames and paths must be uppercase, while historical host-side repository paths should not be renamed just for casing. Add a line-ending rule that contributors must follow `.gitattributes` and keep line-ending normalization separate from functional edits.

Second, add a `.gitattributes` file. Mark common binary artifacts such as `*.EXE`, `*.ZIP`, `*.OBJ`, `*.LIB`, and image/archive formats as binary. Declare text handling for source, batch, PowerShell, Markdown, and reference text. Do not immediately run a full renormalization in the same commit as functional fixes unless the resulting diff is small and obviously mechanical.

Third, fix STUB32C. In `src/stub32a/stub32c.asm`, change the ID32 kernel misc byte from `00111111b` to `10111111b`, and update the adjacent bit-7 comment from `/0=def` to `/1=def`. Rebuild and verify that `BINW\STUB32C.EXE` contains `ID32 BF` at the ID32 block. Also verify that SS reports the STUB32C default as "Ignore DOS/4G API Extension Calls" on.

Fourth, fix SB bind failure handling. In `src/sb/main.c`, keep the existing successful-bind behavior unless a functional reason is found to change it, but move the source deletion so it happens only after `bind_exec` returns success. On any negative return value, close file handles, close the executable context, remove only temporary output if it exists, report the existing error, and leave `newname` intact. Add a focused manual failure test using a disposable copied input and a malformed stub so the failure path can be observed without risking real game files.

Fifth, uppercase DOS-visible names. Update `smoke-dosbox.cmd` to probe `BINW\SVER.EXE` and `BINW\DOS32A.EXE`. Review `README.md`, `ChangeLog`, batch files, distribution scripts or plan text touched by this release, and any generated patch scripts so DOS-facing paths are written as uppercase. Keep source-tree references such as `src/sb/main.c` in their real repository casing.

Sixth, fix stale validation documentation. In `README.md`, change the validation section so it says there is one scripted DOSBox-X smoke test plus required manual compatibility validation for extender changes. The text should not overstate automated coverage.

Seventh, resolve version formatting. The minimum acceptable outcome is a comment near `src/dos32a/dos32a.asm` version word and matching comments in SS/SVER explaining that for major versions 9 and later, a low byte of 10 or greater is interpreted as `minor*10+patch`; therefore `0x1A0B` is `26.1.1` and a future `26.2.0` should be encoded as `0x1A14` if three-part display is desired. If a future two-part version must be encoded as `0x1A02`, then the runtime banner in `src/dos32a/text/client/strings.asm` and `data.asm` must be made conditional so it agrees with SS and SVER.

Eighth, normalize line endings in a separate mechanical step if it is manageable. Run `git ls-files --eol` before the step, run `git add --renormalize .`, inspect `git diff --cached --stat`, and stop if the diff is too broad to review alongside this correctness release. If the diff is acceptable, commit it separately with no logic changes. If it is too broad, narrow the first normalization to active repo-authored files and leave historical `ref/`, `dos32a_800/`, `dos32a-735-src/`, and `tasm5/` material for a later dedicated cleanup.

## Idempotence and Recovery

The build commands are safe to repeat. `build.cmd clean` removes known intermediate and output files and `build.cmd` regenerates `BINW` outputs.

The SB failure test must use disposable copies only. If it fails in an unexpected way, delete only the disposable test directory or files created for that test. Never run destructive tests against `C:\DOSBox-X\DRIVEC` game installs or the only copy of a user executable.

The line-ending normalization step must be isolated. If `git add --renormalize .` stages too many files, unstage it with `git restore --staged .` and revert any unintended working-tree line-ending changes with targeted non-destructive checkout or editor normalization only after confirming those files were not user edits. Do not combine a broad line-ending diff with source logic changes.

If a clean-context review finds a new high or medium issue, pause the release commit, add the issue to this ExecPlan, fix it, rebuild, and rerun review.

# Repair DOS/32A 26.1 DPMI handoff behavior

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `.agent/PLANS.md` from the repository root.

## Purpose / Big Picture

DOS/32A 26.1 must be a trustworthy DOS/4GW replacement, not a set of game-name shortcuts. A user with HDPMI32 loaded should be able to run both bound games, where DOS/32A is embedded in the game executable, and external-launch games, where a game starts a separate `DOS4GW.EXE` replacement. BUILD-engine games need the external HDPMI32 path for linear frame buffer write combining, and id Tech games should not freeze merely because HDPMI32 is loaded.

The visible outcome is a rebuilt `binw\dos32a.exe` whose protected-mode host selection is controlled by the documented DPMI/VCPI detection order and whose external-DPMI setup behaves close to historical DOS/32A 7.x rather than masking problems by recognizing game names.

## Surprises & Discoveries

- Observation: The documentation conflicts with itself about `/DPMITST`.
  Evidence: `dos32a_800/docs/html/user/4.html` says `/DPMITST:ON` looks for a DPMI host first, but `src/dos32a/kernel.asm` and `dos32a_800/docs/html/tech/10.html` define bit 0 as `0=test DPMI/VCPI, 1=VCPI/DPMI /1=default`. The setup UI in `src/ss/menus.c` also displays bit set as `VCPI/DPMI`.

- Observation: Current 26.1 contains a branch-specific HDPMI preference that is not inherited from DOS/32A 7.10, 7.35, or 8.00.
  Evidence: `src/dos32a/text/kernel/detect.asm` has `@@detect_HDPMI`, `src/dos32a/kernel.asm` exports `pm32_hdpmi`, and `src/dos32a/text/client/misc.asm` sets that byte when launched as `DOS4GW.EXE`.

- Observation: The current name-based policy cannot help bound launch modes.
  Evidence: Duke3D in `C:\DOSBox-X\drivec\DUKE3D` has `DUKE3D.EXE` and no external `DOS4GW.EXE`; the game starts DOS/32A from the embedded stub, so checking whether the extender process name is `DOS4GW.EXE` never fires.

- Observation: DOS/32A 7.10/7.35 installed only four protected-mode exception handlers under an external DPMI host.
  Evidence: `dos32a-735-src/src/dos32a/text/client/misc.asm` and `dos32a_800/src/dos32a/text/client/misc.asm` install handlers for exceptions 00h, 06h, 0Dh, and 0Eh after checking `_sys_type == 3`. Current 26.1 installs handlers 00h through 0Eh.

- Observation: The tracked assembly sources preserve mixed historical line endings.
  Evidence: `git ls-files --eol` reports `i/mixed w/mixed` for most touched assembly files. A mechanical CRLF conversion made `git diff --check` report carriage returns as trailing whitespace, so the source must be left in its existing mixed style.

- Observation: The remaining id Tech failure is sound-related.
  Evidence: The user reports that disabling sound makes the same id Tech games work perfectly under HDPMI32 with the repaired DPMITST path.

- Observation: The failure does not appear to be controlled by the BIOS APIC/PIC routing option.

- Observation: The repaired host-selection split is good enough to keep other tested games working.

## Context and Orientation

The repository root is `C:\Users\mbeutler\Projects\dos32a`. The maintained extender source is under `src/dos32a`. Historical source trees are available under `dos32a-735-src/src/dos32a` and `dos32a_800/src/dos32a`.

DPMI means DOS Protected Mode Interface. It is the service API a protected-mode DOS program uses for memory, selectors, interrupts, and real-mode calls. HDPMI32 is an external DPMI host; when DOS/32A chooses it, HDPMI32 performs part of the protected-mode service work. VCPI means Virtual Control Program Interface. It is an older protected-mode interface often provided by memory managers. Linear frame buffer write combining means the CPU or host combines video-memory writes for better throughput; on this hardware it gives a large BUILD-engine frame-rate improvement when the game maps the VESA linear frame buffer through HDPMI32.

There are two relevant launch modes. In a bound launch, the DOS/32A stub is embedded in the game executable, such as `DUKE3D.EXE` or id Tech executables. In an external launch, the game starts a separate extender executable named `DOS4GW.EXE`, such as Redneck Rampage and S2/SR. A correct fix must be compatible with both modes because DOS/4GW supports both.

The main detection code is `src/dos32a/text/kernel/detect.asm`. It sets `pmodetype` to external DPMI, VCPI, XMS, or raw mode before the kernel initializes. The main client setup code is `src/dos32a/text/client/misc.asm`. It installs protected-mode interrupt and exception handlers after the extender has entered protected mode.

## Plan of Work

First, remove the HDPMI32-specific preference byte and helper from normal source. Keep `apply_dos4gw_name_config` only for DOS/4GW compatibility defaults, and remove the call that sets a hidden HDPMI preference. This makes bound and external launch modes use the same detection rules.

Second, make `/DPMITST` behavior explicit and consistent with the code. The kernel bit currently means `0=DPMI first` and `1=VCPI first`. Because the public user documentation says `/DPMITST:ON` means DPMI first, the repair should either change `cfg_env_test` to match the user documentation or document that the setup UI and technical docs define the true behavior. The candidate will favor user-visible command-line behavior: `/DPMITST:ON` should clear the bit so DPMI is tried before VCPI, and `/DPMITST:OFF` should set the bit so VCPI is tried before DPMI.

Third, remove the XMS/IOPL shortcut that skips DPMI and VCPI tests before honoring the configured order. The historical 7.x and 8.00 detection code checks XMS availability, then checks DPMI/VCPI in the configured order, then falls back to raw or XMS operation. That is the behavior needed for HDPMI32 to be selected without a game-name workaround.

Fourth, under an external DPMI host, install only exception handlers 00h, 06h, 0Dh, and 0Eh. Current 26.1 installs 00h through 0Eh, including debug and breakpoint traps, which can intercept game or DPMI-host behavior that historical DOS/32A did not touch. For non-external-DPMI paths, leave the current handler set unchanged in this first candidate to avoid broadening the change.

Fifth, build with `cmd /c build.cmd dos32a`, package the rebuilt `binw\dos32a.exe`, and run the local smoke script with a 10-second timeout. DOSBox-X cannot prove the real-hardware HDPMI32 path, but it can catch build, launch, and immediate-crash regressions.

## Validation and Acceptance

Local acceptance requires a clean build, a passing 10-second DOSBox-X smoke run, no leftover DOSBox-X processes, and a package containing the rebuilt extender plus a manifest.

Real-hardware acceptance requires these observations with HDPMI32 loaded:

1. Bound id Tech executables do not freeze when DOS/32A selects HDPMI32 through normal `/DPMITST:ON` detection.
2. Bound Duke3D is not dependent on an external `DOS4GW.EXE` name rule.
3. Redneck Rampage keeps the linear frame buffer write-combining performance improvement.
4. S2/SR 65k color mode remains working after the selector-pair fix.

If a candidate fails, the result must still be useful: the manifest should make clear whether detection order, exception-vector scope, or a later DPMI service is the likely remaining fault.
